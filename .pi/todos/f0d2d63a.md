## Fix implemented

Added file-based locking using `mkdir` (atomic on both Windows and POSIX) to serialize ID allocation + entity writes.

### Changes

1. **`src/io/files.ts`** — Added `withLock(dir, fn)` utility:
   - Creates `.arad/.lock` directory (atomic acquisition via `mkdir`)
   - Retries every 50ms with 5s timeout
   - Releases lock (removes dir) in `finally` block
   - Stale lock recovery: user can manually delete `.arad/.lock`

2. **`src/commands/add.ts`** — `addCommand` now wraps `createEntity()` in `withLock()`

3. **`src/commands/import.ts`** — `importCommand` now wraps the import loop in `withLock()` (also made async)

4. **`src/commands/validate.ts`** — `promoteCommand` now wraps `performPromote()` in `withLock()` (also made async)

5. **`src/index.ts`** — Updated `action` handlers for `import` and `promote` commands to `async/await`

6. **`tests/concurrency.test.ts`** — New test file with 6 tests:
   - Lock acquisition and release
   - Concurrent access serialization
   - Timeout behavior
   - Parallel `createEntity` produces unique IDs (the actual fix)
   - Demonstrates the bug exists without locking

### Design notes
- Pure logic functions (`createEntity`, `performPromote`) remain synchronous for testability
- Locking is applied at the CLI layer, wrapping the critical section (ID alloc → write)
- `createEntity` is still usable without locking in tests (single-process, no contention)
