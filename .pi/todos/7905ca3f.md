## Context

When an assumption is invalidated, the natural next step is often to create a requirement for the *opposite*. Currently `promote` only works on validated assumptions. The workflow is manual: `add requirement` + `unlink` + `link` — at least 3 steps.

## Example

A-011 "No concurrent writes" was invalidated. The correct follow-up was to create R-031 "Must handle concurrent writes safely" and relink D-034. This required manual steps.

## Proposal

Add an `arad invalidate <id>` command (or a flag on validate) that:
1. Marks the assumption as invalidated
2. Prompts for a new requirement title (or infers the negation)
3. Creates the requirement
4. Re-links all decisions that referenced the old assumption to the new requirement
5. Unlinks the invalidated assumption from those decisions

Could also be a `--derive-requirement` flag on `arad validate <id> --invalidated`.
