## Problem
The `arad add` command's interactive mode uses `readline.createInterface` which doesn't work when stdin is piped (e.g. in scripts or CI). The workaround in the current code reads all stdin at once for piped mode, but the flow is fragile — it counts prompts statically and breaks if prompts are added/removed.

## What happened
During testing, `echo "" | arad add requirement "Title"` would silently fail to create the file. The readline interface would close after the first prompt when stdin was piped, causing subsequent prompts to resolve with empty strings and the entity to never be written.

## Current workaround
The code checks `process.stdin.isTTY` and reads all lines at once for piped input. But this is brittle — if you add a new prompt to the add flow, you must update the line count.

## Better fix
Two options:
1. Make all interactive fields into CLI flags so `arad add` can run fully non-interactively: `arad add requirement "Title" --status=accepted --derived-from=R-001`
2. Skip all interactive prompts when stdin is not a TTY and use defaults

The current flags (`--status`, `--tags`, `--driven-by`) partially solve this but `derived_from` and `conflicts_with` for requirements and `enables`/`supersedes` for decisions still require interactive input even in piped mode.

## Where
`src/commands/add.ts`
