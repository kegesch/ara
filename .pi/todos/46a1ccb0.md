## Problem

`link` and `unlink` commands call `updateEntity` which does a read-modify-write cycle, but neither command wraps its operations in `withLock`. Running multiple link/unlink operations in rapid succession can cause lost writes.

Observed: `unlink D-034 A-011` printed `✓ Removed driven_by` but the file still contained A-011. Likely caused by a race with a concurrent `link` call.

## Fix

Wrap the core logic in `performLink` and `performUnlink` with `withLock`, same as `add` and `promote` already do. The `performLink`/`performUnlink` functions should accept a `dir` parameter and the CLI entry points should call `withLock(process.cwd(), () => performLink(...))`.

## Affected files

- `src/commands/link.ts` — `linkCommand()` and `unlinkCommand()` need `withLock`
