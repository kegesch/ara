## Goal
Detect contradictions between requirements beyond explicit `conflicts_with` declarations.

## Scope
- **Explicit contradictions** already work (the `conflicts_with` field)
- Add **heuristic detection** in `arad check`:
  - **Keyword opposition** — requirements containing opposing terms (e.g., "must encrypt" vs "must not encrypt", "offline" vs "always online", "real-time" vs "batch")
  - **Duplicate detection** — two requirements with very similar titles/body but different IDs (possible unintended duplication)
  - **Status contradictions** — an accepted decision driven by a deprecated/rejected requirement
  - **Orphan chains** — a requirement with no decisions driving from it (requirement exists but nobody acted on it)
- Start simple: keyword opposition pairs + duplicate detection via fuzzy title matching
- Add a configurable opposition dictionary (built-in defaults + user-extensible via `.arad/arad.yaml`)
- Output should be suggestions ("possible contradiction") not assertions

## Why
The whole point of ARAD is finding where requirements conflict or where decisions lack backing. Manual `conflicts_with` is good but insufficient — people forget to declare them.
